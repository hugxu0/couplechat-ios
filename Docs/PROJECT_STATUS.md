# 项目现状

## 产品定位

悄悄话当前是侧载给两位现有用户的共同生活与聊天 App。iOS 客户端直连 `https://hoo66.top`，服务端业务数据存放在 PostgreSQL，媒体文件存放在服务器 `uploads/`。

V2 已确认扩展为可注册、邀请码配对的情侣产品，并要求同账号 iPhone/iPad 同时登录、全量同步。目标产品与迁移路线分别见 `V2_PRODUCT_PLAN.md` 和 `V2_ARCHITECTURE.md`。

## 当前工作树已实现

以下同时包含已上线的 legacy V1 与尚未部署的 V2 候选；生产边界仍是 v1–v10，V2 客户端和 v11–v22 必须经过 macOS CI、隔离迁移/恢复演练后才能发布。

### 账号与同步

- legacy 快捷账号、其他账号登录、注册、创建两人空间、邀请码加入
- 设备绑定 token、有效设备列表、单设备撤销和 iPhone/iPad 同时在线
- 首次启动有界快照，之后通过 REST 分页和 Socket.IO 增量同步
- Sync V2 cursor/ack、持久化 tombstone 和前台约 10 秒跨设备补拉
- 前后台连接恢复、在线状态、已读状态和离线补漏
- 设备端 SQLite 消息缓存，断网可查看已同步内容

### 聊天

- `couple` 公聊与每个账号独立的 `ai` 私聊
- 文字、图片、视频、语音、文件和贴纸；Live Photo 当前只发送静态图，V2 媒体域将补完整配对资源
- 乐观发送、持久化 outbox、失败重试和 clientId 幂等
- 引用回复、两分钟内撤回、消息搜索和定位
- 媒体预览、收藏、缓存、聊天壁纸和互动特效
- Markdown 消息渲染（含表格）与 Mermaid 流程图
- AI 事项确认卡展示、布局缓存与事项归属

### 共享功能

- 纪念日和日期设置
- 个人/共享提醒与备忘；到期通知统一依赖 Bark，共享提醒发双方、私人提醒只发创建者
- 基于本地消息的日/月聊天统计
- 主题色、深浅模式、头像、Bark key 和存储管理
- 共同相册：聊天媒体入册、封面、共同注脚、改名/删除、那年今日
- 共享/私人日历：月视图、编辑、完成、删除和版本冲突
- 服务端持久化大橘：今日双人回应、一次性结算藏品、小窝布置、互动和足迹
- 语音异步转写：失败重试、旧语音补建、人工纠正、搜索和撤回级联

### 大橘 AI

- 公聊通过 `@大橘` 召唤，AI 私聊默认响应每条文字或图片
- OpenAI Agents SDK + 受控 MCP 工具
- 图片理解、联网搜索、来源卡片和事项确认卡
- 六层结构化 Memory、原文证据、上下文摘要和定时维护
- 设置中的 Memory 控制中心：范围/分类/搜索、来源查看、纠正、重要标记、删除和立即整理
- 仅本机开发环境开放的 AI 光路与 Memory 调试页

## 当前限制

- 新注册情侣的聊天、设置、提醒、相册、日历、宠物和基础 AI 回复已经隔离；完整 Agent/MCP 历史检索和自动 Memory 提取仍只对 legacy `xu/si` 开放，新情侣暂用无历史 AI 模式。
- Sync V2 已覆盖当前可变业务域并用前台轮询补拉，但 `sync:available` Socket 唤醒、通用 mutation 去重和 cursor retention/410 重建策略尚未全部接入客户端。
- Memory 控制中心已有分页、版本冲突、跨设备刷新和删除排除；来源消息一键跳回聊天、本地 Memory 离线缓存仍未实现。
- Bark 点击后的页面 deep link 尚未接入。
- 工程已启用 Universal device family、横竖屏、自适应 Tab/侧栏、时光双栏和宠物 60/40 布局；聊天 iPad 主从双栏、照片拖放、完整键盘快捷键与真机矩阵尚未完成。
- 共同相册 MVP 从聊天媒体入册，尚未提供相册内直接拍摄/选择照片上传。
- 日历本身不自动发到期通知；需要通知的事项应使用“提醒”，其 shared/personal Bark 规则已完成。
- Windows 开发机不能本地编译 iOS，必须依赖 GitHub Actions 或 Mac。
- 本地云数据库调试禁止媒体写入，因此上传链路要在生产服务或隔离环境验证。
- 媒体失败重传依赖 outbox 文件；用户主动清理 App 数据后无法继续重传。
- `MessageStore` / `ChatStore` 仍是兼容 facade；新增功能不得继续扩大它们。消息映射/远端分页、Socket 生命周期、领域事件路由和 outbox 串行/文件职责已下沉，上传/ack UI 投影与发送编排仍待继续收敛。

## 维护优先级与残余风险

1. 消息可靠性仍是最高保护项：发送幂等、outbox、同步游标和媒体绑定不能退化。
2. `MessageStore` 和 `ChatStore` 仍承担兼容 facade 职责；新增功能应落到 Repository/Coordinator，不再扩大 facade。
3. V2 ownership、cursor sync、设备级 Bark、相册、日历和宠物基线已经落地；下一优先级是新情侣完整 Agent/Memory 工具迁移、`sync:available` 唤醒和 preference 云同步，不能退回全局字符串所有权。
4. AI 修改必须验证公聊隐私、私聊隔离、原文证据和无模型时的可见兜底。
5. 每次修复后同步更新本文件中的限制，并补充能稳定复现问题的测试。
6. 视觉统一按 V2 五入口推进，同时保护聊天发送、键盘和贴底链路；iPad 必须按可用宽度自适应，不能放大手机界面。

## 已固化的保护边界

以下内容是后续维护必须继续遵守的保护边界，历史任务与验收记录见 `REFACTOR_PLAN.md`：

- 聊天页现有键盘弹起、收起和输入栏跟随效果已达到可用标准，相关重构不得无理由改变动画、时序或底部 inset。
- 聊天顶部材质与文字适配、媒体打开/退出转场、历史同步生命周期、失败媒体重试与删除已经修复，后续必须作为回归保护行为。
- AI 私聊与公聊召唤、Memory、确认卡、记录、提醒、纪念日和互动特效必须保留并纳入回归验证。
- 宠物已经改为服务端权威、双方共同回应与布置藏品的真实系统；奖励结算必须保持事务幂等，客户端不得本地覆盖服务端数值或重复结算。
- 服务端历史是事实源；允许两部设备重新登录或清空本地 SQLite，但上线前必须确认没有仍需保留的待发消息。
