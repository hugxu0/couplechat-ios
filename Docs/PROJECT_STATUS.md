# 项目现状

## 产品定位

悄悄话是仅供两位固定用户使用的私密聊天 App。iOS 客户端直连 `https://hoo66.top`，所有服务端业务数据存放在生产 PostgreSQL，媒体文件存放在服务器 `uploads/`。

账号标识固定为：

| username | 显示名 |
|---|---|
| `xu` | 小旭 |
| `si` | 小偲 |

## 已实现

### 账号与同步

- 账号选择、密码登录、token 持久化和失效核验
- 首次启动有界快照，之后通过 REST 分页和 Socket.IO 增量同步
- 前后台连接恢复、在线状态、已读状态和离线补漏
- 设备端 SQLite 消息缓存，断网可查看已同步内容

### 聊天

- `couple` 公聊与每个账号独立的 `ai` 私聊
- 文字、图片、视频、语音、文件、贴纸和 Live Photo
- 乐观发送、持久化 outbox、失败重试和 clientId 幂等
- 引用回复、两分钟内撤回、消息搜索和定位
- 媒体预览、收藏、缓存、聊天壁纸和互动特效
- Markdown 消息渲染（含表格）与 Mermaid 流程图
- AI 事项确认卡展示、布局缓存与事项归属

### 共享功能

- 纪念日和日期设置
- 个人/共享提醒与备忘，本地通知
- 基于本地消息的日/月聊天统计
- 主题色、深浅模式、头像、Bark key 和存储管理

### 大橘 AI

- 公聊通过 `@大橘` 召唤，AI 私聊默认响应每条文字或图片
- OpenAI Agents SDK + 受控 MCP 工具
- 图片理解、联网搜索、来源卡片和事项确认卡
- 六层结构化 Memory、原文证据、上下文摘要和定时维护
- 仅本机开发环境开放的 AI 光路与 Memory 调试页

## 当前限制

- 大橘宠物页的数值、动作和模型仍是展示占位，不具备持久化宠物状态。
- Bark 点击后的页面 deep link 尚未接入。
- Windows 开发机不能本地编译 iOS，必须依赖 GitHub Actions 或 Mac。
- 本地云数据库调试禁止媒体写入，因此上传链路要在生产服务或隔离环境验证。
- 媒体失败重传依赖 outbox 文件；用户主动清理 App 数据后无法继续重传。
- `MessageStore` / `ChatStore` 仍是兼容 facade；新增功能不得继续扩大它们。消息映射/远端分页、Socket 生命周期、领域事件路由和 outbox 串行/文件职责已下沉，上传/ack UI 投影与发送编排仍待继续收敛。

## 维护优先级与残余风险

1. 消息可靠性仍是最高保护项：发送幂等、outbox、同步游标和媒体绑定不能退化。
2. `MessageStore` 和 `ChatStore` 仍承担兼容 facade 职责；新增功能应落到 Repository/Coordinator，不再扩大 facade。
3. iPad 真机与两台设备同时在线的完整矩阵未执行，后续获得第二台设备时应补测，不需要为此回退本次发布。
4. AI 修改必须验证公聊隐私、私聊隔离、原文证据和无模型时的可见兜底。
5. 每次修复后同步更新本文件中的限制，并补充能稳定复现问题的测试。
6. 视觉统一按「设计系统 → 外围页 → 聊天样式 → facade 收敛」推进；不得借机推倒聊天发送/键盘/贴底链路。

## 已固化的保护边界

以下内容是后续维护必须继续遵守的保护边界，历史任务与验收记录见 `REFACTOR_PLAN.md`：

- 聊天页现有键盘弹起、收起和输入栏跟随效果已达到可用标准，相关重构不得无理由改变动画、时序或底部 inset。
- 聊天顶部材质与文字适配、媒体打开/退出转场、历史同步生命周期、失败媒体重试与删除已经修复，后续必须作为回归保护行为。
- AI 私聊与公聊召唤、Memory、确认卡、记录、提醒、纪念日和互动特效必须保留并纳入回归验证。
- 宠物互动页暂不投入本轮重构，不得以此为由阻塞聊天主链路和其他已完成功能。
- 服务端历史是事实源；允许两部设备重新登录或清空本地 SQLite，但上线前必须确认没有仍需保留的待发消息。
