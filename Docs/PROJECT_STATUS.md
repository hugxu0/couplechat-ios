# 项目现状

## 产品定位

悄悄话是仅供两位固定用户使用的私密聊天 App。iOS 客户端直连 `https://hoo66.top`，所有服务端业务数据存放在生产 PostgreSQL，媒体文件存放在服务器 `uploads/`。

账号标识固定为：

| username | 显示名 |
|---|---|
| `xu` | 小旭 |
| `si` | 小偲 |

## 已实现

### 账号与同步

- 账号选择、密码登录、token 持久化和失效核验
- 首次启动有界快照，之后通过 REST 分页和 Socket.IO 增量同步
- 前后台连接恢复、在线状态、已读状态和离线补漏
- 设备端 SQLite 消息缓存，断网可查看已同步内容

### 聊天

- `couple` 公聊与每个账号独立的 `ai` 私聊
- 文字、图片、视频、语音、文件、贴纸和 Live Photo
- 乐观发送、持久化 outbox、失败重试和 clientId 幂等
- 引用回复、两分钟内撤回、消息搜索和定位
- 媒体预览、收藏、缓存、聊天壁纸和互动特效

### 共享功能

- 纪念日和日期设置
- 个人/共享提醒与备忘，本地通知
- 基于本地消息的日/月聊天统计
- 主题色、深浅模式、头像、Bark key 和存储管理

### 大橘 AI

- 公聊通过 `@大橘` 召唤，AI 私聊默认响应每条文字或图片
- OpenAI Agents SDK + 受控 MCP 工具
- 图片理解、联网搜索、来源卡片和事项确认卡
- 六层结构化 Memory、原文证据、上下文摘要和定时维护
- 仅本机开发环境开放的 AI 光路与 Memory 调试页

## 当前限制

- 大橘宠物页的数值、动作和模型仍是展示占位，不具备持久化宠物状态。
- Bark 点击后的页面 deep link 尚未接入。
- Windows 开发机不能本地编译 iOS，必须依赖 GitHub Actions 或 Mac。
- 本地云数据库调试禁止媒体写入，因此上传链路要在生产服务或隔离环境验证。
- 媒体失败重传依赖 outbox 文件；用户主动清理 App 数据后无法继续重传。

## 当前风险与接手优先级

1. 先在真机复现并记录仍存在的聊天、媒体和前后台连接问题。
2. 优先保护消息可靠性：发送幂等、outbox、同步游标和媒体绑定不能退化。
3. 修复 UI 问题时关注 `ChatViewController.swift`、`MessageStore.swift` 等大文件的职责继续膨胀。
4. AI 修改必须验证公聊隐私、私聊隔离、原文证据和无模型时的可见兜底。
5. 每次修复后同步更新本文件中的限制，并补充能稳定复现问题的测试。

## 完成定义

一个改动至少满足：

- 后端 `npm test` 与 `npm run build` 通过；
- iOS 相关改动通过 SwiftLint、单元测试和 Archive；
- 协议变化已同时更新两端契约与 `Docs/API.md`；
- 数据结构、AI 或部署变化已更新对应文档；
- 不包含密钥、生产数据、媒体副本和构建产物。
